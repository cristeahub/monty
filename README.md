# monty, the head butler

Monty is a small OCaml launcher for running a head pi session that plans work and spins out worker pi sessions in new Ghostty tabs, windows, or splits.
The model does the planning.
Monty handles the glue around Ghostty, `wt`, worktrees, manifests, and `pi` startup commands.

## Requirements

- OCaml with Dune 3.20 or newer.
- Dune package management, with dependencies declared in `dune-project`.
- No opam files are used by this repo.
- `pi` on `PATH`.
- `wt` on `PATH`.
- Ghostty on macOS for normal launching.

## Build

```sh
dune build
```

The repo includes a `dune.lock` generated by Dune package management.
If dependencies change, refresh it with:

```sh
dune pkg lock
dune build
```

## Install

Install the `monty` CLI without opam:

```sh
./install.sh
```

The installer builds with Dune, copies the Monty control room to `~/.local/share/monty`, installs the real binary under that directory, and writes a wrapper at `~/.local/bin/monty`.
It also writes `MONTY_HOME=~/.local/share/monty` and `MONTY_BRANCH_PREFIX=monty` to your shell startup file, such as `~/.zshrc`.
If that file already has a non-Monty-managed `MONTY_HOME` setting, the installer asks before overriding it.
The wrapper pins `MONTY_HOME` and `MONTY_BRANCH_PREFIX` on every invocation, so running `monty` from any directory starts pi in the installed Monty control room and uses the configured worker branch prefix.
Use another prefix with:

```sh
./install.sh --prefix /path/to/prefix
```

Use another Monty home with:

```sh
./install.sh --monty-home /path/to/monty-home
```

Set your worker branch prefix with:

```sh
./install.sh --branch-prefix cto
```

For development, keep the control room in the current checkout and only install the CLI wrapper and binary:

```sh
./install.sh --dev-install --branch-prefix cto
```

Preview the commands without changing anything:

```sh
./install.sh --dry-run
```

## Start the head butler

From this repo, run:

```sh
dune exec -- monty
```

After installation, run:

```sh
monty
```

This changes to the Monty control-room directory and runs:

```sh
pi --name "Monty Head Butler"
```

## Launch one worker

```sh
dune exec -- monty launch \
  --repo /path/to/repo \
  --title "Fix issue 123" \
  --context .monty/runs/run-1/issue-123.md
```

By default Monty creates or reuses a worktree with `wt b <branch>`.
Then it opens Ghostty and runs pi in the worker worktree with the context file passed as an `@file`.

## Launch many workers

```sh
dune exec -- monty launch-many --manifest .monty/runs/run-1/jobs.json
```

Example manifest:

```json
{
  "jobs": [
    {
      "title": "Fix issue 123",
      "repo": "/Users/cristea/code/example",
      "branch": "monty/issue-123",
      "context": ".monty/runs/run-1/issue-123.md"
    }
  ]
}
```

The manifest can also omit `branch`.
Monty then derives a safe branch name from the title using the branch prefix.
The default is `monty`, so `Fix issue 123` becomes `monty/fix-issue-123` for one launch or `monty/01-fix-issue-123` in `launch-many`.
With `--branch-prefix cto` or `MONTY_BRANCH_PREFIX=cto`, the same task becomes `cto/fix-issue-123` or `cto/01-fix-issue-123`.

## Dry run

Use dry-run mode to inspect what Monty would do without creating worktrees or opening Ghostty.

```sh
dune exec -- monty launch-many \
  --terminal dry-run \
  --manifest .monty/runs/run-1/jobs.json
```

## Configuration

The most important options are available as CLI flags.

```sh
--terminal ghostty|dry-run
--target tab|window|split
--worktree always|never
--branch-prefix PREFIX
--pi-command COMMAND
--wt-command COMMAND
--fork SESSION
--home DIR
```

Environment defaults are also supported.

```sh
MONTY_TERMINAL=ghostty
MONTY_TARGET=tab
MONTY_WORKTREE=always
MONTY_BRANCH_PREFIX=cto
MONTY_PI_COMMAND=pi
MONTY_WT_COMMAND=wt
MONTY_HOME=/path/to/monty
```

## Doctor

```sh
dune exec -- monty doctor
```

This checks for `pi`, `wt`, Ghostty, `osascript`, `sdef`, and Dune.
